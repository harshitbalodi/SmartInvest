1. PRODUCT VISION & OVERVIEW
================================================================================

InvestIQ is an AI-powered investment advisory platform for the Indian market.

Target User: Indian retail investors (ages 20-60) who want data-driven 
investment guidance without paying for expensive advisory services.

What InvestIQ Does:
  - Personalized asset allocation advice based on user profile (age, goals, risk)
  - Smart stock screener with 15+ fundamental filters
  - Curated portfolio strategies (10-baggers, dividend kings, compounders)
  - Mutual fund recommendations (active vs passive management)
  - Historical backtesting of strategies with benchmark comparison

What InvestIQ Does NOT Do:
  - Execute trades (information only)
  - Guarantee returns (educational tool)
  - Replace SEBI-registered advisors (always show disclaimers)

NFR-02: RELIABILITY
───────────────────
NFR-02.1  Uptime target                              → 99% (POC)
NFR-02.2  AI provider failover                       → Automatic
          (Gemini → Groq → Ollama)
NFR-02.3  Data staleness tolerance:
          - Fundamentals: 24 hours
          - Stock prices: 5 minutes (market hours)
          - MF NAVs: 24 hours
NFR-02.4  Graceful degradation                       → App works with 
          cached data if external APIs fail

NFR-03: SECURITY
────────────────
NFR-03.1  Password storage       → bcrypt with salt (12 rounds)
NFR-03.2  JWT security           → Access token: 15min, 
                                    Refresh token: 7 days (httpOnly cookie)
NFR-03.3  API rate limiting      → Per-user: 60 req/min
                                    Per-IP: 120 req/min
NFR-03.4  Input validation       → Pydantic schemas on every endpoint
NFR-03.5  CORS                   → Whitelist frontend domain only
NFR-03.6  SQL injection          → Parameterized queries via ORM
NFR-03.7  Secrets management     → .env files, never in source code
NFR-03.8  HTTPS                  → TLS termination at deployment platform

NFR-04: SCALABILITY
───────────────────
NFR-04.1  Stateless backend      → No server-side sessions, JWT-based
NFR-04.2  Cache-first design     → Redis for hot data, reduces DB load
NFR-04.3  Request coalescing     → Dedup concurrent identical API calls
NFR-04.4  Horizontal scaling     → Can add instances behind load balancer
NFR-04.5  DB connection pooling  → Prevent connection exhaustion

NFR-05: MAINTAINABILITY
───────────────────────
NFR-05.1  Clean layered architecture (routes → services → data)
NFR-05.2  Auto-generated Swagger/OpenAPI documentation
NFR-05.3  Unit tests for scoring engine, allocation logic
NFR-05.4  Integration tests for API endpoints
NFR-05.5  Structured JSON logging with correlation IDs
NFR-05.6  Environment-based configuration, no hardcoded values

NFR-06: COMPLIANCE
──────────────────
NFR-06.1  SEBI disclaimer prominent on every advice page
NFR-06.2  No buy/sell order execution — information only
NFR-06.3  Data source attribution (NSE, BSE, AMFI)
NFR-06.4  "Past performance ≠ future results" on all backtest pages

5. TECHNOLOGY STACK & JUSTIFICATION
================================================================================

BACKEND:
────────
  Language:    Python 3.11+
  Framework:   FastAPI (async, auto-docs, Pydantic validation)
  ORM:         SQLAlchemy 2.0 (async) + Alembic (migrations)
  Task Queue:  APScheduler (for nightly CRON jobs)
  Testing:     pytest + httpx (async test client)

  WHY PYTHON OVER NODE.JS:
  - Finance libraries (yfinance, pandas, nse-py, mftool, backtrader)
    are ALL Python-native. No equivalent in Node.js.
  - AI/ML ecosystem (LangChain, scikit-learn) is Python-first.
  - Pandas DataFrames are industry standard for financial data.
  - Using Node.js would require wrapping Python scripts or 
    reimplementing what pandas does from scratch.
  - FastAPI is async and benchmarks close to Express.js performance.

FRONTEND:
─────────
  Framework:   Next.js 14 (React, TypeScript)
  Styling:     Tailwind CSS
  Charts:      Recharts (for equity curves, pie charts, bar charts)
  Tables:      TanStack Table (sortable, filterable stock tables)
  Forms:       React Hook Form + Zod (validation)
  State:       Zustand (lightweight global state)
  HTTP:        Axios / fetch with React Query (caching + auto-refetch)
  Auth UI:     Custom + @react-oauth/google
  Animations:  Framer Motion
  Icons:       Lucide React

DATABASE:
─────────
  Primary DB:  PostgreSQL 15
               (Financial data is relational: stocks→sectors, MFs→AMCs)
               Free tier: Supabase or Neon (500MB) or Render
  Cache:       Redis 7
               (Hot data: prices, screener results, AI responses)
               Free tier: Redis Cloud (30MB) or Upstash (10K cmds/day)

DEPLOYMENT:
───────────
  Backend:     Render.com free tier / Railway.app
  Frontend:    Vercel free tier
  Database:    Supabase free tier (500MB PostgreSQL)
  Redis:       Upstash free tier (10K commands/day)

AI PROVIDERS:
─────────────
  Primary:     Google Gemini (15 RPM, 1M tokens/day FREE)
  Fallback:    Groq (Llama 3.1 70B, 30 RPM FREE)
  Offline:     Ollama + Mistral 7B (runs on local machine)

DEVOPS:
───────
  Containers:  Docker + Docker Compose (local development)
  CI/CD:       GitHub Actions (lint, test, deploy)
  Monitoring:  Structured logging (JSON format)

6. AI PROVIDER STRATEGY
================================================================================

Multi-Provider AI Gateway with Automatic Failover:

         User Request
              │
              ▼
      ┌───────────────┐
      │  AI GATEWAY   │  ← Provider-agnostic abstraction layer
      │  (You build   │
      │   this)       │
      └───────┬───────┘
              │
       ┌──────┼──────────┐
       ▼      ▼          ▼
    Gemini   Groq    Ollama/Local
    (Primary) (Fallback) (Offline)

Cost Analysis:
  Google Gemini Free Tier:
    - 15 requests per minute
    - 1,000,000 tokens per day
    - Models: Gemini 1.5 Flash (fast), Gemini 1.5 Pro (quality)
    - More than enough for POC with 100 users

  Groq Free Tier:
    - 30 requests per minute
    - Llama 3.1 70B model (excellent quality)
    - Ultra-fast inference (~500 tokens/sec)
    
  Ollama (Local):
    - Completely free
    - Needs 16GB RAM minimum
    - Models: Mistral 7B, Llama 3.1 8B
    - Speed: ~5-15 tokens/sec on CPU
    - Quality: Decent for basic advice

Interview Talking Point:
  "I built a provider-agnostic AI gateway with automatic failover. 
   If Gemini rate-limits, it falls back to Groq, and can run fully 
   offline with Ollama for air-gapped demos."

WHY THIS ARCHITECTURE:

Decision                      | Why
──────────────────────────────┼────────────────────────────────────────
Monolith (not microservices)  | POC stage. Faster to build, deploy, 
                              | debug. Logical separation via modules.
                              | Can extract to microservices later.
PostgreSQL (not MongoDB)      | Financial data is relational. Stocks 
                              | have sectors, MFs have AMCs. ACID needed 
                              | for user data. Free on Supabase.
Redis (not in-memory dict)    | Need TTL, shared across instances, 
                              | pub/sub for cache invalidation. 
                              | Industry standard.
Nightly ETL (not real-time)   | Free APIs don't support WebSocket. 
                              | EOD data sufficient for fundamentals. 
                              | Saves 99% of API calls.
AI Gateway pattern            | Provider-agnostic. Can switch/failover 
                              | without changing business logic. Shows 
                              | design maturity.
FastAPI (not Flask/Django)    | Async native, auto-generates OpenAPI 
                              | docs, Pydantic validation, best 
                              | performance among Python frameworks.

14. BACKTESTING ENGINE
================================================================================

BACKTESTING FLOW:
  1. User selects strategy + time period + initial investment
  2. System goes back to start_date
  3. Runs the screener with HISTORICAL data as of that date
  4. Picks top 10 stocks, equal weight allocation
  5. Simulates holding until next rebalance date (quarterly)
  6. Recalculates portfolio value based on actual price changes
  7. Rebalances: runs screener again, new top 10
  8. Repeats until end_date
  9. Calculates final metrics

METRICS CALCULATED:
  - Final portfolio value
  - CAGR (Compound Annual Growth Rate)
  - Nifty 50 CAGR (benchmark) over same period
  - Alpha (strategy CAGR minus benchmark CAGR)
  - Max Drawdown (worst peak-to-trough decline)
  - Sharpe Ratio (risk-adjusted return)
  - Win Rate (% of quarters with positive returns)
  - Best/Worst stock picks

EXAMPLE OUTPUT:
  Strategy: Ten Baggers
  Period: Jan 2020 → Mar 2025 (5 years)
  Initial: ₹1,00,000
  Final: ₹4,85,000
  CAGR: 37.2%
  Nifty CAGR: 14.8%
  Alpha: +22.4%
  Max Drawdown: -32% (Mar 2020)
  Sharpe: 1.8
  Win Rate: 72%

DISCLAIMER: Survivorship bias warning. Past performance ≠ future results.

15. CACHING STRATEGY (MULTI-LAYER)
================================================================================

THREE CACHE LAYERS:

Layer 1: Redis (Hot Cache)
  - Stock prices: TTL 60 seconds
  - MF NAVs: TTL 24 hours
  - Fundamentals: TTL 24 hours
  - Screener results: TTL 1 hour
  - AI responses: TTL 12 hours
  - Strategy picks: TTL 6 hours

Layer 2: PostgreSQL (Warm Cache)
  - Full fundamental data for all ~5000 stocks
  - All MF NAVs and performance data
  - Updated via nightly ETL
  - Historical data stored permanently

Layer 3: External APIs (Cold — Only on Cache Miss)
  - Called only when Redis AND PostgreSQL miss
  - Or when forced refresh by admin
  - Rate-limited and throttled

REQUEST FLOW:
  1. Check Redis → HIT → Return (< 5ms)
  2. Miss → Check PostgreSQL → HIT → Return + Populate Redis (< 50ms)
  3. Miss → Call External API → Return + Populate Both (< 2000ms)

DATA FRESHNESS BY TYPE:
  Stock Price (live)     → Cache 60s    → On-demand + cache
  Day's OHLC             → Cache 24h    → EOD batch job
  Fundamentals           → Cache 24h    → Nightly CRON
  Financials             → Cache 7 days → After results season
  Company Info           → Cache 30 days → Monthly refresh
  MF NAVs                → Cache 24h    → Nightly CRON
  Market Indices         → Cache 30s    → On-demand + cache

16. REQUEST COALESCING PATTERN
================================================================================

PROBLEM:
  50 users request TCS stock data at the exact same moment.
  Without dedup: 50 external API calls → rate limited → expensive
  With dedup: 1 API call, 49 users wait for the same result

SOLUTION (Singleflight Pattern):
  - Maintain a dictionary of "in-flight" requests
  - When a request comes in, check if someone is already fetching it
  - If yes: wait for their result (don't make another API call)
  - If no: make the API call, share result with everyone waiting

  This is the same pattern used by Go's singleflight package 
  and CDNs like Cloudflare.

INTERVIEW TALKING POINT:
  "I implemented request coalescing to deduplicate concurrent API calls. 
   If 100 users request the same stock simultaneously, only 1 external 
   API call is made, reducing costs by 99%."

  Register → Hash password (bcrypt, 12 rounds) → Store in PostgreSQL
  Login → Verify password → Issue JWT pair:
    Access Token:  15 min expiry, sent in Authorization header
    Refresh Token: 7 days expiry, sent in httpOnly secure cookie

  On expiry: Frontend calls /auth/refresh → New access token issued

GOOGLE OAUTH FLOW:
  Frontend → Google Sign-In → Receives ID Token
  Backend → Verify ID token with Google's public keys
  Backend → Create user (if new) or find existing user
  Backend → Issue JWT pair (same as above)

TOKEN PAYLOAD:
  {
    "sub": "user_uuid",
    "email": "user@email.com",
    "iat": issued_at_timestamp,
    "exp": expiry_timestamp,
    "type": "access" | "refresh"
  }

SECURITY MEASURES:
  - Passwords: bcrypt with 12 salt rounds
  - Access tokens: Short-lived (15 min)
  - Refresh tokens: httpOnly cookie (not accessible via JavaScript)
  - Rate limiting on /auth endpoints: 10 req/min (prevent brute force)
  - CORS: Whitelist only frontend domain

22. DESIGN DECISION LOG (INTERVIEW READY)
================================================================================

These are the key technical decisions and the reasoning behind each.
Use these as talking points in interviews and portfolio reviews.

Decision 1: Python (FastAPI) over Node.js (NestJS)
  Why: 80% of backend is data processing + AI integration.
  Finance libraries (yfinance, pandas, nse-py, mftool, backtrader)
  exist ONLY in Python. Node.js would require wrapping Python or
  reimplementing what pandas already does.

Decision 2: Monolith over Microservices
  Why: POC stage — ship fast, debug easily. Logical separation via
  modules (auth, advisor, screener, data). Can extract services later
  when scaling demands it. Premature microservices = wasted effort.

Decision 3: Multi-provider AI Gateway with Failover
  Why: Cost optimization (Gemini free → Groq free → Ollama local).
  Provider-agnostic abstraction means swapping AI providers requires
  zero business logic changes. Shows resilience thinking.

Decision 4: Nightly ETL Pipeline over Real-time Streaming
  Why: Free APIs don't support WebSocket/streaming. EOD data is
  sufficient for fundamental analysis (PE, ROE don't change intraday).
  Pre-fetching at market close means ZERO external API calls during
  peak user hours.

Decision 5: Request Coalescing (Singleflight Pattern)
  Why: If 100 users request the same stock simultaneously, only 1
  external API call is made. Others await the same future/promise.
  Reduces external API costs by up to 99%. Same pattern used by
  Go's singleflight package and CDNs like Cloudflare.

Decision 6: Multi-layer Cache (Redis + PostgreSQL + API)
  Why: Redis (hot, <10ms) → PostgreSQL (warm, <50ms) → External API
  (cold, <2s). 95% of requests served from cache. Cache-aside pattern
  with TTL-based invalidation. Handles thundering herd via coalescing.

Decision 7: PostgreSQL over MongoDB/NoSQL
  Why: Financial data is inherently relational (stocks belong to
  sectors, MFs belong to AMCs, users have portfolios with holdings).
  Need ACID guarantees for user financial data. Free on Supabase.

Decision 8: Composite Scoring over Simple Pass/Fail Filtering
  Why: A stock with ROE 14.8% shouldn't be excluded when filter is
  ROE > 15%. Percentile-based scoring gives a 0-100 score. Multiple
  factors weighted per strategy. More nuanced than binary filtering.

Decision 9: JWT (Access + Refresh) with httpOnly Cookies
  Why: Access token (15min) in memory for API calls. Refresh token
  (7 days) in httpOnly cookie prevents XSS theft. Refresh rotation
  prevents replay attacks. Stateless = horizontally scalable.

Decision 10: Active vs Passive MF Scoring with Different Weights
  Why: Active funds should be scored on alpha, consistency, fund
  manager tenure. Passive funds should be scored on tracking error
  and expense ratio (the ONLY things that matter for index funds).
  One scoring model cannot serve both styles.
